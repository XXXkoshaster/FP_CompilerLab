## Введение

### Цель сравнения

Данный проект исследует, как три большие языковые модели (LLM) справляются с одной и той же задачей — реализацией интерпретатора функционального языка программирования на F#. Сравниваются архитектурные решения, полнота реализации, качество кода и обработка ошибок.

### Техническое задание

Реализовать интерпретатор функционального языка программирования на F# со следующими возможностями:

- **Базовые выражения**: целые числа, булевы значения, переменные
- **Функции**: лямбда-выражения, применение функций, замыкания
- **Привязки**: `let`-выражения, рекурсивные привязки (`letrec`)
- **Управление потоком**: условные выражения (`if`)
- **Структуры данных**: списки с операциями `cons`, `car`/`head`, `cdr`/`tail`
- **Встроенные функции**: арифметика (`+ - * /`), сравнения (`= < > <= >=`)
- **Функции высшего порядка**: `map`, `filter`, `fold`
- **Ввод-вывод**: вывод значений на экран

Эталонная реализация представлена в директории `exampl_project/` — она использует собственный синтаксис и включает полный набор тестов (18 тестовых сценариев).

## Описание реализаций

### Claude — `FunLang_claude/`

**Язык**: F# (.NET 8.0), без внешних зависимостей

**Архитектура**: классический пайплайн `Tokenizer → Parser → Interpreter`

```
Lang.fs (12 строк)         — AST: Expression, Environment (список мутабельных фреймов)
Tokenizer.fs (48 строк)    — ручной токенизатор (char list → Token list)
Parser.fs (29 строк)       — рекурсивный спуск (Token list → Expression list)
Interpreter.fs (181 строка) — tree-walking интерпретатор
Program.fs (30 строк)      — точка входа, чтение .scm файла
```

**Итого**: ~300 строк кода в 5 файлах

**Ключевые особенности**:
- Полноценный Scheme-диалект с S-выражениями
- Мутабельные фреймы окружения (`Frame = Map<string, Expression ref> ref`) — позволяют реализовать `define` и `letrec` с мутацией
- Разделение на `Function` (обычные функции) и `Special` (специальные формы: `if`, `let`, `lambda`, `define`, `begin`) — аргументы специальных форм не вычисляются до передачи
- Поддержка строк и файлового ввода-вывода (`read-file`, `write-file`, `append-file`, `file-exists?`)
- Комментарии в исходном коде на русском языке
- Режим работы: исполнение `.scm` файлов из командной строки
- 6 готовых примеров программ в `examples/`

---

### DeepSeek — `FunLang_deepseek/`

**Язык**: F# (.NET 10.0), зависимость: FParsec 1.1.1

**Архитектура**: `FParsec Parser → AST → Evaluator` с отдельным слоем значений

```
AST.fs (10 строк)          — AST: Expr (Number, Boolean, Symbol, Lambda, Apply, If, Let, LetRec)
Parser.fs (78 строк)       — FParsec-парсер с forward references для взаимной рекурсии
Evaluator.fs (67 строк)    — eval/apply, отдельный тип Value (VNumber, VBoolean, VClosure, VPrimitive, VPair, VNil)
Primitive.fs (63 строки)   — встроенные функции и начальное окружение
Repl.fs (24 строки)        — интерактивный REPL
Examples.fs (30 строк)     — примеры программ как строковые константы
Program.fs (10 строк)      — точка входа
```

**Итого**: ~282 строки кода в 7 файлах

**Ключевые особенности**:
- Чёткое разделение AST (`Expr`) и значений (`Value`) — семантически правильный подход
- `VPair`/`VNil` для представления списков как cons-пар (классический Lisp-подход)
- Замыкания хранят `ref` на окружение — используется для патчинга в `letrec`
- Логические операции: `not`, `and`, `or`
- Режим работы: интерактивный REPL

---

### GPT — `FunLang_gpt/`

**Язык**: F# (.NET 10.0), зависимость: FParsec 1.1.1

**Архитектура**: `FParsec Parser → AST → Evaluator`

```
AST.fs (10 строк)          — AST (идентичен DeepSeek)
Parser.fs (73 строки)      — FParsec-парсер (почти идентичен DeepSeek)
Evaluator.fs (54 строки)   — eval/apply (почти идентичен DeepSeek)
Builtins.fs (26 строк)     — встроенные функции
Program.fs (25 строк)      — REPL с командой :q для выхода
```

**Итого**: ~188 строк кода в 5 файлах

**Ключевые особенности**:
- Минималистичная реализация — только ядро языка
- Хорошая абстракция для встроенных операций (`numBinOp`, `cmpBinOp`, `expectNumber`)
- Режим работы: интерактивный REPL с командой `:q`

## Сравнительная таблица

| Критерий | Claude | DeepSeek | GPT |
|----------|:------:|:--------:|:---:|
| **Полнота реализации ТЗ** | 4/5 | 3/5 | 2/5 |
| **Читаемость и структура кода** | 3/5 | 4/5 | 4/5 |
| **Обработка ошибок** | 4/5 | 3/5 | 3/5 |
| **Расширяемость** | 4/5 | 4/5 | 5/5 |
| **Наличие тестов** | 1/5 | 1/5 | 1/5 |
| **Документация** | 5/5 | 2/5 | 1/5 |
| **Объём кода** | ~300 строк | ~282 строки | ~188 строк |

### Пояснения к оценкам

**Полнота реализации ТЗ**:
- *Claude (4/5)*: реализованы все базовые требования + бонусные возможности (строки, файловый ввод-вывод, `begin`, `display`). Не хватает встроенных `map`/`filter`/`fold` (но их можно реализовать через `letrec` на уровне языка).
- *DeepSeek (3/5)*: базовые конструкции реализованы. Есть `cons`/`car`/`cdr`/`list`/`null?`, логические операции. Нет встроенных `map`/`filter`/`fold` и нет поддержки `display`/`print` с форматированием.
- *GPT (2/5)*: реализовано только ядро — 7 встроенных операций (`+ - * / = < >`). Нет списков (`cons`, `car`, `cdr`, `list`, `null?`), нет `print`/`display`, нет логических операций.

**Читаемость и структура кода**:
- *Claude (3/5)*: монолитный `Interpreter.fs` (181 строка) — все встроенные функции объявлены внутри одного `makeGlobalEnv`. Использование `ref` повсеместно усложняет понимание.
- *DeepSeek (4/5)*: хорошее разделение на модули, чистое отделение AST от значений. Однако однострочные определения примитивов в `Primitive.fs` снижают читаемость.
- *GPT (4/5)*: самый компактный и чистый код. Выделение `expectNumber`, `numBinOp`, `cmpBinOp` в `Builtins.fs` — хороший паттерн для расширения.

**Обработка ошибок**:
- *Claude (4/5)*: единообразная функция `malformed` для всех ошибок формы, информативные сообщения при несоответствии аргументов.
- *DeepSeek (3/5)*: `failwith`/`failwithf` с описательными сообщениями, проверка арности в `apply`.
- *GPT (3/5)*: аналогично DeepSeek, но меньше контекста в сообщениях об ошибках.

**Расширяемость**:
- *Claude (4/5)*: добавление новых функций — одна запись в `makeGlobalEnv`. Специальные формы требуют написания отдельной функции.
- *DeepSeek (4/5)*: новые примитивы — добавление в `initialEnv` в `Primitive.fs`.
- *GPT (5/5)*: самая модульная архитектура. `Builtins.fs` легко расширяется благодаря абстракциям `numBinOp`/`cmpBinOp`.

**Наличие тестов**:
- Ни одна из трёх реализаций не содержит тестов. Для сравнения, эталонный `exampl_project` содержит 18 тестов в `InterpreterTests.fs`.

**Документация**:
- *Claude (5/5)*: подробный `README.md` на русском языке с описанием синтаксиса, таблицей встроенных функций, чек-листом реализованных возможностей.
- *DeepSeek (2/5)*: `Examples.fs` с 4 программами-примерами, но без README.
- *GPT (1/5)*: документация отсутствует полностью.

## Аналитическая часть

### Сильные и слабые стороны

#### Claude

**Сильные стороны**:
- Наиболее полная реализация из трёх — единственный проект с поддержкой строк, файлового ввода-вывода и комментариев в исходном языке
- Ручной токенизатор и парсер без зависимостей — полный контроль над обработкой синтаксиса
- Единственный проект с режимом выполнения файлов (остальные — только REPL)
- Подробная документация и 6 готовых примеров программ

**Слабые стороны**:
- Монолитный `Interpreter.fs` — все встроенные функции определены в одном месте через замыкания
- Интенсивное использование `ref` для мутабельности среды — идиоматически нетипично для F#
- Нет разделения AST и значений — `Expression` служит и для представления синтаксического дерева, и для результатов вычислений (`Function`, `Special`, `Dummy`)

#### DeepSeek

**Сильные стороны**:
- Архитектурно наиболее правильное разделение: `Expr` (AST) ≠ `Value` (результат вычисления)
- Cons-пары (`VPair`/`VNil`) — классическое Lisp-представление списков
- Логические операции (`and`, `or`, `not`) — единственный проект с полной булевой логикой
- Красивая рекурсивная функция `valueToString` для отображения вложенных списков

**Слабые стороны**:
- Однострочные определения примитивов в `Primitive.fs` — тяжело читать и поддерживать
- REPL не сохраняет окружение между выражениями (`repl env` всегда передаёт один и тот же `env`)
- Нет файлового режима выполнения — невозможно запустить программу из файла

#### GPT

**Сильные стороны**:
- Самый компактный и чистый код (188 строк)
- Лучшая модульность встроенных функций — `numBinOp`/`cmpBinOp` исключают дублирование
- Легче всего расширять благодаря абстрактным помощникам

**Слабые стороны**:
- Критически неполная реализация — отсутствуют списки, ввод-вывод, `print`
- Нет документации
- Только 7 встроенных операций — язык практически непригоден для написания содержательных программ

**Замечание**:
Также важно отметить, что импирическим путем выявлено нецелесобразность использования DeepSeek при написании осмысленного кода. DeepSeek постоянно выдавал код с мелкими логическими ошибками и несоответствиями типов, которые не выявлялись при первой компиляции. Потребовалось огромное количество времени на «вайбдебаг», чтобы заставить интерпретатор просто заработать.

### Интересные и неожиданные решения

1. **Идентичный код GPT и DeepSeek**: файлы `AST.fs`, `Parser.fs` и `Evaluator.fs` у GPT и DeepSeek практически совпадают — одинаковые имена переменных, одинаковая структура, одинаковые комментарии. Это может указывать на общие паттерны в обучающих данных обеих моделей или на генерацию из одного промпта с разной степенью детализации.

2. **Claude выбрал другой дизайн**: в отличие от GPT и DeepSeek, Claude не использует FParsec и реализовал полностью самостоятельный пайплайн (Tokenizer + Parser). Это единственная реализация с типом `Special` для ленивых (не-вычисляющих аргументы) форм — архитектурно более зрелое решение.

3. **Мутабельная среда у Claude**: использование `Map<string, Expression ref> ref` для фреймов окружения — нетипично для функционального подхода, но позволяет естественно реализовать `define` с побочными эффектами и `letrec` с мутацией.

4. **Файловый ввод-вывод у Claude**: `read-file`, `write-file`, `append-file`, `file-exists?` — ни одна другая модель не вышла за рамки чистых вычислений.

---

### Типичные ошибки и паттерны LLM

1. **Игнорирование тестов**: ни одна модель не сгенерировала тестов — все три проекта имеют 0 тестов, в то время как эталонный проект содержит 18. LLM склонны генерировать «рабочий» код, но не сопровождать его верификацией.

2. **Шаблонность решений GPT и DeepSeek**: почти полное совпадение кода двух «разных» моделей ставит под вопрос разнообразие генерируемых решений.

3. **Преждевременное обрубание функциональности**: GPT остановился на минимальном ядре, не реализовав даже списки — ключевую структуру данных функционального языка.

4. **Отсутствие документации**: только Claude сгенерировал README. DeepSeek и GPT не предоставили ни описания синтаксиса, ни инструкций по запуску.

## Общий вывод

### Рейтинг

1. **Claude** — наиболее полная и практически применимая реализация. Единственный проект, который можно использовать «из коробки» для запуска программ: есть файловый режим, документация, примеры, строки и ввод-вывод. Архитектурное решение с `Special`-формами демонстрирует более глубокое понимание семантики Lisp-подобных языков.

2. **DeepSeek** — архитектурно наиболее грамотная реализация с правильным разделением AST и значений. Достаточная база встроенных функций для написания простых программ. Однако отсутствие документации и файлового режима ограничивают практическое использование.

3. **GPT** — самый чистый и расширяемый код, но критически неполная реализация. Без списков и ввода-вывода язык не выполняет даже минимальных требований ТЗ. Фактически это каркас, а не готовый интерпретатор.

### Когда какая модель предпочтительнее

| Задача | Рекомендуемая модель |
|--------|---------------------|
| Полная реализация «под ключ» с документацией | **Claude** |
| Архитектурно чистый прототип для дальнейшей разработки | **DeepSeek** |
| Минимальный каркас для самостоятельного расширения | **GPT** |

```bash

# GPT — интерактивный REPL (выход: :q)
cd FunLang_gpt && dotnet run

# Эталонный проект — запуск тестов
cd exampl_project && dotnet run
```
